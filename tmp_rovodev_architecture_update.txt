
## Security & Compliance Architecture (December 2024)

### Google Play Compliance Implementation

**Version:** 1.0.21+25  
**Status:** Gold Standard Implementation

#### Permission Architecture

**Removed Unused Permissions:**
- Bluetooth stack (5 permissions) - Not used by app
- Location services (2 permissions) - Not used by app  
- Samsung Health SDK (2 permissions) - App uses Health Connect instead

**Current Permission Model:**
- Health Connect (13 permissions) - Core fitness tracking
- Camera (1 permission) - Nutrition photo capture with rationale dialog
- Activity Recognition - Step counting
- Notifications (Android 13+) - Streak reminders
- Storage - Temporary meal photo storage

#### Privacy & Data Architecture

**Privacy Policy Infrastructure:**
- In-app policy: `lib/screens/legal/privacy_policy_screen.dart`
- Public URL: https://streaker-website.pages.dev/privacy.html
- Contact: novatrient@gmail.com, Bangalore, India
- Compliance: Google Play Data Safety, GDPR, COPPA

**Data Safety Declaration:**
- Personal Info: Name, email, user IDs
- Health & Fitness: Steps, heart rate, calories, sleep, weight, workouts
- Photos: Meal images (nutrition analysis only)
- App Activity: Interactions, search history
- Analytics: Firebase (anonymized)

**Third-Party Data Sharing:**
- Supabase: Backend storage (encrypted)
- Google AI/Gemini: Meal photo analysis (not stored)
- Firebase: Analytics & Cloud Messaging

#### Account Deletion Architecture

**UI Layer:**
- Component: `lib/widgets/delete_account_dialog.dart`
- Location: Profile > Settings > Delete Account
- Safety: Checkbox + "DELETE" text confirmation

**Service Layer:**
- Method: `SupabaseService.deleteUserAccount()`
- Deletion scope: 9 tables (nutrition_entries, weight_entries, workout_sessions, workout_templates, achievements_progress, daily_nutrition_summary, streaks, user_devices, profiles)
- Storage: Deletes profile photos from Supabase Storage
- Auth: Signs out user after deletion
- Note: Auth user remains in auth.users (requires service role key for full deletion)

#### Data Export Architecture (GDPR)

**Service:** `lib/services/data_export_service.dart`
**Format:** JSON with 2-space indentation
**Export Contents:**
- Profile information
- Nutrition entries with statistics
- Workout sessions and templates
- Weight history with trend analysis
- Achievement progress
- Streak data
- Daily summaries (90 days)

**Statistics Calculated:**
- Total records
- Date range (first/last entry)
- Days of data
- Total calories tracked
- Total workouts
- Weight change

**Sharing:** Via share_plus package (email, drive, etc.)

#### Permission Rationale System

**Architecture:**
- Component: `lib/widgets/permission_rationale_dialog.dart`
- Trigger: Before system permission dialog
- Pattern: Show rationale → User accepts → Request system permission
- User control: Can decline without blocking functionality

**Camera Rationale Flow:**
```
User taps camera → Check permission status → 
If denied → Show rationale dialog → 
User clicks "Continue" → Request system permission → 
Grant/Deny → Proceed accordingly
```

**Rationale Content:**
- Clear explanation of usage
- List of specific use cases
- Optional alternative methods
- Settings reminder

#### Security Infrastructure

**Network Security:**
- Config: `android/app/src/main/res/xml/network_security_config.xml`
- Policy: `cleartextTrafficPermitted="false"`
- Enforcement: HTTPS-only, system certificates only
- Protection: Prevents MITM attacks

**Code Obfuscation (ProGuard/R8):**
- Rules: `android/app/proguard-rules.pro`
- Enabled: `isMinifyEnabled = true`, `isShrinkResources = true`
- Coverage: Flutter, Supabase, Firebase, OkHttp, Health Connect, Play Core
- Benefits: Reverse engineering protection, optimized build size, debug log removal

**Consent Management (Optional):**

- Storage: SharedPreferences (persistent)
- Status: Created but not integrated (future enhancement)
